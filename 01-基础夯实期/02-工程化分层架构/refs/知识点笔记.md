## 一、APIRouter 路由模块化

### 1.1 为什么需要 APIRouter

单文件 `main.py` 随着接口增多会迅速膨胀到上千行。APIRouter 可以按**业务模块**将路由拆分到独立文件中，每个模块只关注自己的接口。

### 1.2 创建子路由

```python
# app/api/v1/tasks.py
from fastapi import APIRouter

router = APIRouter(prefix="/tasks", tags=["任务管理"])

@router.get("/")
def list_tasks():
    return [{"id": 1, "title": "示例任务"}]

@router.post("/", status_code=201)
def create_task():
    return {"id": 2, "title": "新任务"}
```

### 1.3 挂载到主应用

```python
# app/main.py
from fastapi import FastAPI
from app.api.v1 import tasks

app = FastAPI(title="任务管理系统")
app.include_router(tasks.router)
```

### 1.4 APIRouter 参数详解

| 参数 | 用途 | 示例 |
|---|---|---|
| `prefix` | 路由前缀，所有路径自动添加 | `prefix="/tasks"` → `/tasks/`, `/tasks/{id}` |
| `tags` | 文档分组标签 | `tags=["任务管理"]` |
| `dependencies` | 全局依赖（该路由下所有接口生效） | `dependencies=[Depends(verify_token)]` |
| `responses` | 全局响应定义 | `responses={404: {"description": "Not found"}}` |

```python
router = APIRouter(
    prefix="/tasks",
    tags=["任务管理"],
    dependencies=[Depends(get_db)],
    responses={404: {"description": "任务不存在"}},
)
```

### 1.5 多级路由挂载

```python
# app/api/v1/__init__.py
from fastapi import APIRouter
from app.api.v1 import tasks, users, items

router = APIRouter(prefix="/v1")
router.include_router(tasks.router)
router.include_router(users.router)
router.include_router(items.router)

# app/main.py
from app.api.v1 import router as v1_router
app.include_router(v1_router)
```

最终 URL 结构：

| 定义位置 | 最终路径 |
|---|---|
| `v1` 前缀 `/v1` | — |
| `tasks` 前缀 `/tasks` | `/v1/tasks/` `/v1/tasks/{id}` |
| `users` 前缀 `/users` | `/v1/users/` `/v1/users/{id}` |

### 1.6 APIRouter 中的路径去重

子路由中定义的路径会拼接到 `prefix` 之后。注意路径中的 `/` 处理：

```python
router = APIRouter(prefix="/tasks")

@router.get("/")      # 匹配 /tasks/
@router.get("/list")  # 匹配 /tasks/list
@router.get("/{id}")  # 匹配 /tasks/{id}
```

## 二、分层架构设计与目录规范

### 2.1 为什么需要分层

分层架构的核心思想是**关注点分离** (Separation of Concerns)：

- **路由层** → 只做 URL 分发，不写业务逻辑
- **业务层** → 只做核心逻辑，不知道 HTTP 细节
- **数据层** → 只做数据存取，不关心业务规则

带来的工程价值：

| 优势 | 说明 |
|---|---|
| **职责清晰** | 每层只做一件事，降低认知负担 |
| **改动隔离** | 修改数据层不影响路由层，反之亦然 |
| **可测试性** | 每层可独立测试，mock 成本低 |
| **团队协作** | 多人并行开发不同层，冲突少 |

### 2.2 官方推荐目录结构

```
project/
├── app/
│   ├── __init__.py
│   ├── main.py                  # FastAPI 应用创建、中间件、异常处理器注册
│   ├── config.py                # 配置管理（pydantic-settings）
│   ├── api/
│   │   ├── __init__.py
│   │   └── v1/
│   │       ├── __init__.py      # 汇总 v1 所有路由
│   │       ├── tasks.py         # /tasks 接口
│   │       └── users.py         # /users 接口
│   ├── models/                  # SQLAlchemy ORM 模型
│   │   ├── __init__.py
│   │   ├── task.py
│   │   └── user.py
│   ├── schemas/                 # Pydantic 请求/响应模型
│   │   ├── __init__.py
│   │   ├── task.py
│   │   └── user.py
│   ├── services/                # 业务逻辑层
│   │   ├── __init__.py
│   │   ├── task_service.py
│   │   └── user_service.py
│   ├── dependencies/            # FastAPI 依赖注入
│   │   ├── __init__.py
│   │   └── auth.py
│   └── utils/                   # 工具函数
│       └── __init__.py
├── tests/
├── alembic/                     # 数据库迁移
└── pyproject.toml
```

### 2.3 各层职责速查

| 层级 | 目录 | 核心职责 | 依赖 |
|---|---|---|---|
| **路由层** | `api/v1/` | URL 分发、HTTP 方法、状态码、参数校验 | 调用 Service |
| **业务层** | `services/` | 核心业务逻辑、多表操作、事务协调 | 调用 Model / Repository |
| **数据模型层** | `models/` | ORM 表映射（SQLAlchemy） | 数据库 |
| **数据模式层** | `schemas/` | Pydantic 请求/响应定义 | — |
| **依赖层** | `dependencies/` | 认证、分页、数据库会话等公共依赖 | — |
| **配置层** | `config.py` | 环境变量、数据库连接串等配置 | — |

### 2.4 分层后的完整调用链路

```
请求
  │
  ▼
api/v1/tasks.py          ← 路由层：URL 分发，参数校验
  │  Depends(get_db)
  ▼
services/task_service.py  ← 业务层：核心逻辑
  │
  ├─ models/task.py       ← 数据层：ORM 操作
  └─ schemas/task.py      ← 数据层：模型序列化
  │
  ▼
响应返回
```

### 2.5 分层后的代码示例

```python
# ----- schemas/task.py -----
from pydantic import BaseModel, Field

class TaskCreate(BaseModel):
    title: str = Field(..., min_length=1, max_length=50)
    priority: int = Field(default=1, ge=1, le=5)

class TaskResponse(TaskCreate):
    id: int
    model_config = {"from_attributes": True}


# ----- services/task_service.py -----
from app.schemas.task import TaskCreate, TaskResponse
from app.models.task import Task

class TaskService:
    def __init__(self, db):
        self.db = db

    def create(self, data: TaskCreate) -> TaskResponse:
        record = Task(**data.model_dump())
        # ... 持久化逻辑
        return TaskResponse.model_validate(record)


# ----- api/v1/tasks.py -----
from fastapi import APIRouter, Depends
from app.schemas.task import TaskCreate, TaskResponse
from app.services.task_service import TaskService
from app.dependencies import get_db

router = APIRouter(prefix="/tasks", tags=["任务管理"])

@router.post("/", response_model=TaskResponse, status_code=201)
def create_task(data: TaskCreate, db=Depends(get_db)):
    service = TaskService(db)
    return service.create(data)


# ----- main.py -----
from fastapi import FastAPI
from app.api.v1 import router as v1_router

app = FastAPI(title="任务管理系统")
app.include_router(v1_router)
```

### 2.6 分层原则速记

| 原则 | 说明 |
|---|---|
| **上层依赖下层** | 路由 → 业务 → 数据，不反向依赖 |
| **同层不互调** | 两个 Service 之间通过上层编排，不直接调用 |
| **下层不知上层** | Service 不知道 HTTP、不知道路由 |
| **每层只做一件事** | 路由只做分发，业务只做逻辑，数据只做存取 |

## 三、全局异常处理器

### 3.1 为什么需要全局异常处理

默认 422 错误返回格式与业务返回不一致，需统一：

```json
// 默认返回
{"detail": [{"loc": ["body", "title"], "msg": "field required", "type": "value_error.missing"}]}

// 统一后返回
{"code": 422, "message": "请求参数校验失败", "errors": [{"field": "title", "msg": "字段必填"}]}
```

### 3.2 自定义异常类

```python
class AppException(Exception):
    """业务异常基类"""
    def __init__(self, code: int, message: str, detail: dict | None = None):
        self.code = code
        self.message = message
        self.detail = detail

class NotFoundError(AppException):
    def __init__(self, resource: str = "资源"):
        super().__init__(code=404, message=f"{resource}不存在")

class ValidationError(AppException):
    def __init__(self, message: str = "请求参数校验失败"):
        super().__init__(code=422, message=message)

class AuthError(AppException):
    def __init__(self, message: str = "未授权访问"):
        super().__init__(code=401, message=message)
```

### 3.3 注册全局异常处理器

```python
from fastapi import FastAPI, Request
from fastapi.responses import JSONResponse
from fastapi.exceptions import RequestValidationError

app = FastAPI()

# 处理自定义业务异常
@app.exception_handler(AppException)
async def app_exception_handler(request: Request, exc: AppException):
    return JSONResponse(
        status_code=exc.code,
        content={
            "success": False,
            "code": exc.code,
            "message": exc.message,
            "detail": exc.detail,
        },
    )

# 覆盖 FastAPI 内置的 422 校验异常
@app.exception_handler(RequestValidationError)
async def validation_exception_handler(request: Request, exc: RequestValidationError):
    return JSONResponse(
        status_code=422,
        content={
            "success": False,
            "code": 422,
            "message": "请求参数校验失败",
            "errors": exc.errors(),
        },
    )

# 兜底：捕获所有未处理异常
@app.exception_handler(Exception)
async def global_exception_handler(request: Request, exc: Exception):
    return JSONResponse(
        status_code=500,
        content={
            "success": False,
            "code": 500,
            "message": "服务器内部错误",
        },
    )
```

### 3.4 异常处理器执行顺序

```
请求 → 参数校验失败 → validation_exception_handler (422)
     → 接口函数抛出 AppException → app_exception_handler
     → 其他未处理异常 → global_exception_handler (500 兜底)
```

### 3.5 注册时机

异常处理器应在**应用创建时立即注册**，放在 `main.py` 中：

```python
app = FastAPI(title="任务管理系统")

# 1. 注册异常处理器（优先——确保请求生命周期早期就绪）
register_exception_handlers(app)

# 2. 注册中间件
app.add_middleware(...)

# 3. 挂载路由
app.include_router(v1_router)
```

## 四、中间件与洋葱模型

### 4.1 什么是中间件

中间件是**在请求到达路由之前**和**响应返回客户端之前**执行的钩子函数，可以无侵入地实现横切关注点（日志、计时、CORS 等）。

### 4.2 中间件标准写法

```python
import time
from fastapi import Request

@app.middleware("http")
async def process_time_middleware(request: Request, call_next):
    # 请求前：记录开始时间
    start_time = time.perf_counter()

    # 调用下一个中间件或路由
    response = await call_next(request)

    # 响应后：计算耗时，注入响应头
    process_time = time.perf_counter() - start_time
    response.headers["X-Process-Time"] = str(round(process_time, 4))
    return response
```

### 4.3 洋葱模型执行流程

```
                    ┌─────────────────────────┐
                    │  请求进入 (Request)       │
                    │                          │
                    ▼                          │
          ┌──────────────────┐                 │
          │  中间件 1 (请求前) │                 │
          │  - 记录开始时间    │                 │
          └──────┬───────────┘                 │
                 ▼                             │
        ┌──────────────────┐                   │
        │  中间件 2 (请求前) │                   │
        │  - 日志记录       │                   │
        └──────┬───────────┘                   │
               ▼                               │
          ┌──────────┐                         │
          │  路由处理    │                       │
          │  (业务逻辑)  │                       │
          └──────┬───┘                          │
                 ▼                             │
        ┌──────────────────┐                   │
        │  中间件 2 (响应后) │                   │
        └──────┬───────────┘                   │
               ▼                               │
          ┌──────────────────┐                 │
          │  中间件 1 (响应后) │                 │
          │  - 注入响应头      │                 │
          └──────┬───────────┘                 │
                 ▼                             │
                    ┌──────────────────────────┘
                    │  响应返回 (Response)
                    ▼
```

**关键特征**：
- 请求按注册顺序**依次进入**
- 响应按注册顺序**反向返回**（后进先出）
- 每个中间件通过 `call_next` 控制是否继续

### 4.4 `call_next` 的本质

```python
@app.middleware("http")
async def my_middleware(request: Request, call_next):
    # request — 当前 HTTP 请求对象
    # call_next — 指向下一个中间件或路由处理函数的引用

    response = await call_next(request)  # 放行请求
    return response                      # 处理响应
```

- `call_next` 是一个 **async 可调用对象**，接收 `Request` 返回 `Response`
- 不调用 `call_next` → 请求被拦截（可用于鉴权失败直接返回）
- 可以在 `call_next` 前后各插入逻辑

### 4.5 常用中间件示例

```python
# 1. CORS 中间件（官方提供）
from fastapi.middleware.cors import CORSMiddleware

app.add_middleware(
    CORSMiddleware,
    allow_origins=["http://localhost:5173"],  # 允许的前端域名
    allow_methods=["*"],                      # 允许的 HTTP 方法
    allow_headers=["*"],                      # 允许的请求头
    allow_credentials=True,                   # 允许携带 Cookie
)


# 2. 请求耗时统计
@app.middleware("http")
async def timing_middleware(request: Request, call_next):
    start = time.perf_counter()
    response = await call_next(request)
    elapsed = time.perf_counter() - start
    response.headers["X-Process-Time"] = f"{elapsed:.3f}s"
    return response


# 3. 请求日志记录
@app.middleware("http")
async def log_requests(request: Request, call_next):
    logger.info(f"→ {request.method} {request.url.path}")
    response = await call_next(request)
    logger.info(f"← {response.status_code}")
    return response
```

### 4.6 `@app.middleware` vs `app.add_middleware`

| 方式 | 适用场景 | 示例 |
|---|---|---|
| `@app.middleware("http")` | 自定义简单逻辑，直接写在 main.py | 耗时统计、日志 |
| `app.add_middleware(Cls)` | 官方/三方中间件，需传配置参数 | CORS、GZip、HTTPS 重定向 |

## 五、结构化日志体系

### 5.1 为什么不用 print

| | `print` | `logging` |
|---|---|---|
| 分级控制 | ❌ 无法区分级别 | ✅ DEBUG / INFO / WARNING / ERROR / CRITICAL |
| 格式控制 | ❌ 固定输出 | ✅ 自定义时间、级别、位置信息 |
| 输出目标 | ❌ 仅控制台 | ✅ 控制台 + 文件 + 远程服务 |
| 生产可用 | ❌ 性能差、无切割 | ✅ 支持滚动、异步、按天切割 |

### 5.2 五级日志速查

| 级别 | 数值 | 适用场景 |
|---|---|---|
| `DEBUG` | 10 | 开发调试，排查细节（生产环境通常关闭） |
| `INFO` | 20 | 正常流程记录（请求到达、任务完成） |
| `WARNING` | 30 | 潜在问题，但未影响功能（配置即将过期） |
| `ERROR` | 40 | 业务错误，部分功能失效（数据库连接失败） |
| `CRITICAL` | 50 | 严重故障，系统不可用（磁盘空间耗尽） |

### 5.3 基础配置

```python
import logging

logging.basicConfig(
    level=logging.INFO,
    format="%(asctime)s | %(levelname)-8s | %(name)s:%(lineno)d | %(message)s",
    datefmt="%Y-%m-%d %H:%M:%S",
)

logger = logging.getLogger(__name__)

# 使用
logger.info("服务启动成功")
logger.error("数据库连接失败", exc_info=True)  # exc_info 输出完整堆栈
```

### 5.4 三个核心组件

```
Logger     → 记录器：应用程序的日志入口
Handler    → 处理器：决定日志输出到哪里（控制台 / 文件 / 网络）
Formatter  → 格式化器：决定日志的最终呈现格式
```

```python
import logging

# 1. 创建 Logger
logger = logging.getLogger("app")
logger.setLevel(logging.DEBUG)

# 2. 创建 Handler
console_handler = logging.StreamHandler()          # 输出到控制台
file_handler   = logging.FileHandler("app.log")    # 输出到文件

console_handler.setLevel(logging.INFO)
file_handler.setLevel(logging.DEBUG)

# 3. 创建 Formatter（统一格式）
formatter = logging.Formatter(
    "%(asctime)s | %(levelname)-8s | %(name)s | %(message)s"
)
console_handler.setFormatter(formatter)
file_handler.setFormatter(formatter)

# 4. 组装到 Logger
logger.addHandler(console_handler)
logger.addHandler(file_handler)
```

### 5.5 生产级日志封装

```python
# app/utils/logger.py
import logging
import sys
from pathlib import Path

LOG_DIR = Path("logs")
LOG_DIR.mkdir(exist_ok=True)


def setup_logger(name: str = "app") -> logging.Logger:
    logger = logging.getLogger(name)
    logger.setLevel(logging.DEBUG)

    # 统一格式
    fmt = logging.Formatter(
        "%(asctime)s | %(levelname)-8s | %(name)s:%(lineno)d | %(message)s",
        datefmt="%Y-%m-%d %H:%M:%S",
    )

    # 控制台输出（INFO 及以上）
    console = logging.StreamHandler(sys.stdout)
    console.setLevel(logging.INFO)
    console.setFormatter(fmt)

    # 文件输出（DEBUG 及以上，按天切割，保留 30 天）
    file = logging.handlers.TimedRotatingFileHandler(
        LOG_DIR / "app.log",
        when="midnight",        # 每天午夜切割
        backupCount=30,         # 保留 30 天
        encoding="utf-8",
    )
    file.setLevel(logging.DEBUG)
    file.setFormatter(fmt)

    logger.addHandler(console)
    logger.addHandler(file)
    return logger


logger = setup_logger()


# 在接口中使用
from app.utils.logger import logger

@router.get("/tasks")
def list_tasks():
    logger.info("查询任务列表")
    try:
        ...
    except Exception as e:
        logger.error(f"查询失败: {e}", exc_info=True)
        raise
```

### 5.6 FastAPI 日志中间件集成

```python
import time
from fastapi import Request
from app.utils.logger import logger

@app.middleware("http")
async def logging_middleware(request: Request, call_next):
    start = time.perf_counter()
    logger.info(f"→ {request.method} {request.url.path}")

    response = await call_next(request)

    elapsed = time.perf_counter() - start
    logger.info(
        f"← {request.method} {request.url.path} "
        f"[{response.status_code}] {elapsed:.3f}s"
    )
    return response
```

### 6.2 main.py 标准骨架

```python
from contextlib import asynccontextmanager
from fastapi import FastAPI
from fastapi.middleware.cors import CORSMiddleware
from app.api.v1 import router as v1_router
from app.core.exceptions import register_exception_handlers
from app.utils.logger import logger


@asynccontextmanager
async def lifespan(app: FastAPI):
    """管理应用生命周期"""
    logger.info("应用启动中...")
    # 初始化资源（数据库连接池等）
    yield
    logger.info("应用关闭中...")
    # 清理资源


app = FastAPI(
    title="任务管理系统",
    version="1.0.0",
    lifespan=lifespan,
)

# ⭐ 注册顺序有讲究

# 1. 异常处理器（最先注册，确保全生命周期可用）
register_exception_handlers(app)

# 2. 中间件
app.add_middleware(
    CORSMiddleware,
    allow_origins=["*"],
    allow_methods=["*"],
    allow_headers=["*"],
)

# 3. 路由挂载（最后）
app.include_router(v1_router)
```
