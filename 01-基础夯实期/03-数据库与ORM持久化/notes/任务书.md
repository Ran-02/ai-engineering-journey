# 第3周周末实战任务书：任务管理系统·数据库持久化版

## 一、任务说明
- 项目目标：在第二周工程化分层项目基础上，将内存存储替换为 SQLite + SQLAlchemy 持久化存储，落地 ORM 核心能力
- 前置基础：已完成第二周工程化分层改造，掌握分层架构、参数校验、异常处理
- 存放路径：`01-基础夯实期/03-数据库与ORM持久化/code/`
- 核心要求：业务功能完全兼容无退化，数据可持久化保存，会话生命周期规范，事务边界清晰

## 二、核心任务清单
### 1. 环境与数据库基础配置
1. 安装核心依赖：`sqlalchemy`
2. 新增 `db` 目录，创建数据库核心配置文件
3. 完成 SQLAlchemy 三大核心组件初始化：
   - Engine：数据库连接引擎，指定 SQLite 连接地址
   - SessionLocal：会话工厂，负责创建数据库会话
   - Base：声明式基类，所有数据库模型的父类
4. 实现 `get_db` 数据库会话依赖，通过 yield 管理会话生命周期，请求结束自动关闭连接

### 2. 数据库模型定义
1. 在 `db/models.py` 中定义 `TaskDB` 数据库模型
2. 映射字段：`task_id`（主键、自增）、`title`、`description`、`priority`
3. 指定表名 `tasks`，明确字段类型与约束
4. 区分数据库模型（DB Model）与接口模型（Schema）的职责边界，两层模型分离设计

### 3. 全接口 CRUD 数据库改造
1. 将所有内存列表操作全部替换为数据库会话操作
2. 五个核心接口改造：
   - 创建任务：新增数据到数据库，提交事务
   - 查询列表：分页查询数据库，返回总数与列表
   - 查询单条：按主键查询单条数据
   - 更新任务：查询到数据后修改字段，提交事务
   - 删除任务：删除指定数据，提交事务
3. 所有写操作（增/改/删）必须显式 commit，查询操作无需提交
4. 操作异常时自动回滚事务，避免脏数据

### 4. 双层模型数据转换
1. 给 Pydantic 响应模型增加 `model_config = {"from_attributes": True}` 配置
2. 实现 ORM 数据库对象自动序列化为接口响应模型
3. 保持接口返回格式与之前完全一致，实现前后端协议无感知升级
4. 确保数据库敏感字段不会泄露到接口返回中

### 5. 项目结构升级与分层优化
1. 完整目录结构升级，新增 db 层，职责边界清晰
2. 保留原有全局异常、中间件、日志体系，功能不退化
3. 数据库相关逻辑全部收敛在 db 层，路由层不直接操作数据库
4. 启动项目时自动创建数据表，无需手动执行 SQL

### 6. 全量回归测试与提交
1. 正常场景全覆盖：验证增删改查功能与改造前完全一致
2. 持久化验证：重启服务后数据依然存在，不丢失
3. 异常场景验证：参数校验、404异常、统一错误格式全部生效
4. 事务验证：构造异常场景，确认数据不会部分写入
5. 整理代码，补充注释，按规范提交至 GitHub 对应目录

## 三、验收标准
✅ 数据持久化生效，重启服务后数据不丢失
✅ 三大组件职责清晰：Engine 负责连接、Session 负责操作、Base 负责映射
✅ 数据库会话通过依赖注入管理，请求结束自动关闭，无连接泄露
✅ 增删改操作显式提交事务，异常自动回滚
✅ 双层模型分离：DB 模型面向数据库、Schema 模型面向接口，职责不混杂
✅ 响应模型支持 ORM 对象自动转换，接口返回格式统一
✅ 原有工程化能力全部保留：全局异常、中间件、日志、参数校验
✅ 路由层不直接操作数据库，数据访问逻辑收敛清晰
✅ 启动项目自动创建数据表，无需额外手动操作
✅ 所有边界场景正常，无未捕获异常，无事务脏数据

## 五、运行方式
1. 安装依赖：`pip install fastapi uvicorn sqlalchemy`
2. 启动服务：`uvicorn app.main:app --reload`
3. 接口文档：http://127.0.0.1:8000/docs
4. 数据存储：项目根目录自动生成 `task.db` SQLite 数据库文件
